5. The Subject (corner silhouette, live-processed)

LARGE

Now: Not implemented. No DOM element, no asset, no code path. Empty state is just #drop-overlay text.

Spec — "The Subject": "small monochrome silhouette of a dancing figure, looping forever in a 3-second loop … Game-Boy-screen-style framed window (~120×120 px) pinned to the top-right corner of the workspace at all times … being processed by the user's current chain in real time."

Asset prep

  • Source three default loops: Dancer, Smoke, Bloom. Format: silhouette (dark figure on light, or inverse — pick one). 3-second loop.
  • Decide format: looped .webm / .mp4 (small, clean, hardware-decoded) vs animated WebGL / Canvas2D procedural. Confirm with user before sourcing assets — this is the biggest open decision.
  • If video files: ensure seamless loop, no audio track, <500 KB target each

DOM + visual frame

  • Add a fixed-position #subject element pinned top-right of the canvas area, ~120×120 px
  • Game-Boy-style frame: simple inset border, slightly tinted background, no skeuomorphism per spec
  • Caps monospace label below: THE SUBJECT
  • Inside the frame: a small <canvas> that displays the chain output for the Subject

Pipeline integration (the hard part)

  • Decide architecture: (a) run a second instance of the entire chain into the Subject's small canvas every frame, OR (b) render the main chain to a texture and downsample. Spec implies the Subject is processed independently of the user's source — so (a) is correct: the Subject's loop is the input texture, not the user's video.
  • Refactor existing per-effect modules (voronoi.js, cellular.js, ascii.js, etc.) so they can run against any input texture + any output FBO size. Audit current modules: many use glContext singletons that may assume one canvas. This may force a small rendering refactor — flag for user before starting.
  • Subject must update on every knob turn / effect swap, even with no user source loaded
  • Performance budget: target the Subject pass <1 ms/frame on integrated GPU. 120×120 is small, but every effect runs twice now.

Interactions

  • Click → maximize the Subject into the main canvas as the active source (useful when no user file is loaded). Decide: does this replace the user's loaded source, or override it temporarily?
  • Right-click → context menu / popover with three picks: Dancer, Smoke, Bloom. Persists per session in localStorage.
  • Default loop on first open: spec doesn't specify — pick Dancer.

Empty-state behavior

  • When no user source: Subject still runs in the corner. Main canvas shows drop hint (already exists).
  • All knobs / effect picks operate on the Subject, providing live feedback before commit.

6. Bottom Source bar

MEDIUM

Now: Source controls live in the left sidebar OSC section: Upload Video + Open Camera buttons, file-status text, Speed as a 4-button discrete toggle group (1× / 2× / 3× / 4×). Loop is hardcoded on the <video> element. No play/pause, reverse, trim, frame-hold.

Spec — "Source controls": filename · play/pause · loop toggle · reverse toggle · speed knob (0.25×–4×) · trim handles on timeline · frame hold button. Always exposed at bottom.

Layout

  • Add a new bottom bar element #source-bar, full-width, fixed to bottom of #app
  • Restructure #app grid so canvas area shrinks vertically to accommodate the bar (decide height, e.g. ~60 px)
  • Move filename / file-status display from sidebar to the new bar (left-aligned)
  • Decide: keep the Upload / Open Camera buttons in the sidebar OR move to the bar. Spec says "Replace source by drag/drop anywhere on the workspace" — implies upload buttons are de-emphasized. Confirm with user.

Transport controls (new)

  • Play / Pause button — toggles video.paused
  • Loop toggle — currently hardcoded; expose as toggleable, default ON for video
  • Reverse toggle — flips playback direction. Implementation note: HTML5 video.playbackRate doesn't reliably support negative values across browsers. Likely needs a frame-by-frame reverse driver (read currentTime backwards on each rAF). Flag complexity to user.
  • Speed knob — replace 4-button discrete group with a continuous knob, 0.25× to 4×, default 1×. Reuse existing knob component (data-knob pattern in DOM).
  • Frame Hold button — pauses video and locks the current frame as a still. Distinct from regular Pause? Spec says "freezes on the current frame; useful for treating a single frame as a still" — likely just Pause + a visual lock indicator. Confirm interpretation.

Trim handles + timeline (decision needed)

  • Spec puts trim handles on a timeline ABOVE the source bar (in the center column). User's in-scope list says "Bottom Source bar" — clarify whether trim handles are part of this work item or a separate later one. Default assumption: defer trim/timeline to a later item; only the bar itself is in scope here.

Empty / image / camera states

  • Empty: bar shows no source, transport controls disabled
  • Image source: filename + image tag, transport controls disabled (Frame Hold has no meaning since the frame is already held)
  • Webcam: filename = webcam, Pause/Loop/Reverse/Speed/Trim/Frame-Hold all disabled (live feed only per spec)
